# 重新审视指令的结构

虽然我们已经进行到了流水线的程度，但是由于过度依赖AI生成代码，对指令Trace的理解依然不够明确。要想完全理解Trace驱动模拟器的工作原理，必须重新审视指令Trace的结构。审视指令Trace就是审视从Spike日志到二进制Trace的打包处理脚本，以及模拟器如何使用这些字段，这反映出整个数据流链条的全貌。

RISC-V 指令根据编码格式分为六大类型：**R 型、I 型、S 型、B 型、U 型、J 型**。这六种编码格式与模拟器中定义的 OpType 枚举（IntAlu、Load、Store、Branch、Jump、System）之间存在映射关系。下文逐一分析每种指令类型在日志中的原始表达、二进制Trace中的字段填充规则、以及模拟器流水线中的处理差异。

## 一、RISC-V 六种指令编码格式回顾

| 指令格式 | 典型用途 | 操作码关键位 (inst[6:0]) | 有 rs1 | 有 rs2 | 有 rd |
|---------|---------|------------------------|--------|--------|-------|
| **R 型** | 寄存器-寄存器运算 | `0x33` (OP) / `0x3B` (OP-W) | ✔ | ✔ | ✔ |
| **I 型** | 立即数运算/Load/JALR/System | `0x13`/`0x03`/`0x67`/`0x73` | ✔ | ✗ | ✔ |
| **S 型** | 存储指令 | `0x23` | ✔ | ✔ | ✗ |
| **B 型** | 条件分支 | `0x63` | ✔ | ✔ | ✗ |
| **U 型** | LUI / AUIPC | `0x37` / `0x17` | ✗ | ✗ | ✔ |
| **J 型** | JAL | `0x6F` | ✗ | ✗ | ✔ |

> 注：JALR 虽然在编码格式上属于 I 型（有 rs1、有 rd），但在功能上属于跳转，因此映射到 OpType::Jump。

## 二、指令 OpType 的分类逻辑

在 `parse_spike_log.py` 中，`get_op_type()` 函数通过机器码的最低 7 位（opcode）决定类型归属：

```
opcode = inst & 0x7F
```

| opcode | 映射 OpType | 包含指令 |
|--------|-----------|---------|
| `0x33` (OP) / `0x3B` (OP-W) | **IntAlu** | add, sub, and, or, slt, sll 等 R 型运算 |
| `0x13` (OP-IMM) / `0x1B` (OP-IMM-W) | **IntAlu** | addi, andi, slli 等 I 型立即数运算 |
| `0x37` (LUI) / `0x17` (AUIPC) | **IntAlu** | U 型指令 |
| `0x03` (LOAD) | **Load** | lw, lb, lh, lbu, lhu |
| `0x23` (STORE) | **Store** | sw, sb, sh |
| `0x63` (BRANCH) | **Branch** | beq, bne, blt, bge, bltu, bgeu |
| `0x6F` (JAL) | **Jump** | jal |
| `0x67` (JALR) | **Jump** | jalr |
| `0x73` (SYSTEM) | **System** | ecall, ebreak 等 |

可见 OpType 并非一对一映射 RISC-V 编码格式——**IntAlu 跨越了 R 型、I 型和 U 型三种编码格式**。这是合理的，因为从 Trace 驱动的视角看，这三种格式的指令在流水线行为上完全一致：它们都在 EX 阶段产生计算结果、在 WB 阶段写回寄存器。

## 三、各指令类型的 Trace 生成全流程

### 通用规则

所有指令的 Trace 记录都是 32 字节二进制块，结构如下：

| 偏移 | 字段 | 类型 | 来源 |
|-----|------|------|------|
| +0 | pc | u64 | 取指行 PC |
| +8 | inst | u32 | 取指行机器码 |
| +12 | rs1 | u8 | 从机器码 bit[19:15] 提取 |
| +13 | rs2 | u8 | 从机器码 bit[24:20] 提取 |
| +14 | rd | u8 | 从机器码 bit[11:7] 提取 |
| +15 | op_type | u8 | 由 opcode 映射得到 |
| +16 | mem_addr | u64 | 从提交行解析（仅 Load/Store 有效） |
| +24 | rd_val | u64 | 从提交行解析（仅写寄存器指令有效） |

但具体哪些字段**真正有意义**，因指令类型而异。部分字段会被强制清零（如 S 型/B 型指令的 rd），部分字段虽然填了值但在模拟器中不被使用。

---

### 1. R 型指令（IntAlu）

**示例指令：`add x11, x12, x13`**

Spike 日志中的表达：
```
core   0: 0x0000000080000070 (0x00d605b3) add     a1, a2, a3
core   0: 3 0x0000000080000070 (0x00d605b3) x11 0x0000000000000037
```

对机器码 `0x00d605b3` 解码：
- opcode = `0x33` → OP_INTALU
- rd = (0x00d605b3 >> 7) & 0x1F = 0x0B = x11 (a1)
- rs1 = (0x00d605b3 >> 15) & 0x1F = 0x0C = x12 (a2)
- rs2 = (0x00d605b3 >> 20) & 0x1F = 0x0D = x13 (a3)

二进制 Trace 字段：

| 字段 | 值 | 说明 |
|-----|-----|------|
| rs1 | 12 (x12) | 从 bit[19:15] 正常提取 |
| rs2 | 13 (x13) | 从 bit[24:20] 正常提取 |
| rd | 11 (x11) | 从 bit[11:7] 正常提取 |
| mem_addr | 0 | Load/Store 以外无意义 |
| rd_val | 0x37 | 来自提交行 `x11 0x...0037` |

流水线行为：
- **EX 阶段**：`alu_result = rd_val`（直接使用 Trace 提供的真值）
- **WB 阶段**：`registers[11] = alu_result`
- 两个源操作数 rs1/rs2 都会参与转发网络，ID 阶段通过 `forward()` 获取最新值

---

### 2. I 型指令（IntAlu 子类——立即数运算）

**示例指令：`addi a1, a0, 20`**

Spike 日志中的表达：
```
core   0: 0x0000000080000004 (0x01450593) addi    a1, a0, 20
core   0: 3 0x0000000080000004 (0x01450593) x11 0x000000000000001e
```

机器码 `0x01450593`：
- opcode = `0x13` → OP_INTALU

与 R 型的核心差异：

| 字段 | 值 | 说明 |
|-----|-----|------|
| rs1 | 10 (x10/a0) | 从 bit[19:15] 提取 |
| **rs2** | **0** | **I 型无 rs2 字段，强制清零** |
| rd | 11 (x11/a1) | 正常提取 |

关键规则：在 `parse_spike_log.py` 中，当 opcode 为 `0x13`（OP-IMM）时，rs2 被**强制设为 0**。这是因为 I 型指令的 bit[24:20] 编码的是立即数而非寄存器索引，不能作为依赖追踪的依据。二进制 Trace 中 rs2=0，转发网络遇到 x0 会直接返回 0，不会误触发数据依赖。

流水线行为与 R 型 IntAlu 完全相同。

---

### 3. U 型指令（IntAlu 子类——LUI / AUIPC）

**示例指令：`lui t0, 0x10000`**

Spike 日志中的表达：
```
core   0: 0x0000000080000000 (0x10000297) lui     t0, 0x10000
core   0: 3 0x0000000080000000 (0x10000297) x5 0x0000000010000000
```

| 字段 | 值 | 说明 |
|-----|-----|------|
| rs1 | **0** | U 型无 rs1 字段，强制清零 |
| rs2 | **0** | U 型无 rs2 字段，强制清零 |
| rd | 5 (x5/t0) | 正常提取 |

关键规则：当 opcode 为 `0x37` 或 `0x17` 时，rs1 和 rs2 **都强制清零**。U 型指令不读取任何源寄存器，因此在依赖追踪中完全透明。

流水线行为与 R 型 IntAlu 相同——通过 rd_val 写回寄存器。

---

### 4. Load 指令（I 型——访存加载）

**示例指令：`lw a1, 0(a2)`**

Spike 日志中的表达：
```
core   0: 0x0000000080000240 (0x00062583) lw      a1, 0(a2)
core   0: 3 0x0000000080000240 (0x00062583) x11 0x0000000000000037
```

机器码 `0x00062583`：
- opcode = `0x03` → OP_LOAD
- rs1 = 12 (x12/a2) —— 基地址寄存器
- rd = 11 (x11/a1)
- rs2 = **0**（I 型无 rs2）

注意 Spike 的提交行对于 Load 指令**只输出 rd 的写回值**，不输出访存地址。这是因为 Spike 的 `--log-commits` 只记录**架构状态的改变**——Load 指令的架构效果是"将内存值读入 rd"，不改变内存内容，因此 Spike 只记录 `x rd value`；而 Store 指令的架构效果是"修改内存"，因此 Spike 记录 `mem addr written_value`。

Load 指令的内存地址在 Spike 日志中不可直接获取。地址由 `rs1 + 符号扩展立即数` 计算得出，其中：
- `rs1` 来自 bit[19:15]——**寄存器编号**可在指令编码中取得
- **立即数** `imm` 编码在 bit[31:20]（I 型格式）：`inst[31:20]` 共 12 位，需要符号扩展
- `rs1` 的**值**在日志解析阶段不可直接获取（Spike 未输出通用寄存器的值，只有 rd 写回的值）

因此，要获取 Load 的完整访存地址，脚本需要模拟执行之前的所有指令来追踪 `rs1` 的值，或者在 Spike 一端打补丁输出访存地址。当前 `parse_spike_log.py` 对 Load 的 `mem_addr` 保持为 0。

> **是否需要补全 Load 的 `mem_addr`？**
>
> 补全 `mem_addr` 本身是可行的——在解析脚本中维护寄存器状态表追踪 `rs1` 的值，或在模拟器 EX 阶段从 `inst` 解码立即数后与 `rs1_val` 相加即可。但关键在于：**在什么模拟目标下这信息才是必要的？**
>
> `mem_addr` 的意义在于"这条 Load 访问了哪个内存地址"，其使用场景包括：
> - 缓存层级模拟（命中/缺失判定依赖精确地址）
> - 内存别名分析（乱序执行中 Store-to-Load forwarding 需要比较地址）
> - TLB / 页表行为建模
> - 预取策略建模
>
> 而在当前关注的多发射场景中，核心问题是同一周期内多条指令之间的**寄存器数据依赖**检测（RAW、WAW）。这个检测完全依赖 `rs1`、`rs2`、`rd` 三个字段——它们已经齐全。Load 的访存地址不参与多发射的依赖分析，内存序约束对于顺序多发射也并非瓶颈。
>
> **结论：在只关注多发射、不做完整访存模拟的阶段，补全 Load 的 `mem_addr` 并非当前必要的需求，可以搁置。**

二进制 Trace 字段：

| 字段 | 值 | 说明 |
|-----|-----|------|
| rs1 | 12 (x12) | 基址寄存器（参与依赖追踪） |
| rs2 | 0 | I 型无 rs2 |
| rd | 11 (x11) | 目标寄存器 |
| mem_addr | 0 (或有效地址) | 取决于 Spike 日志是否输出 |
| rd_val | 0x37 | **加载到 rd 的内存值** |

流水线行为——这是最特殊的一类：
- **EX 阶段**：`alu_result = mem_addr`（地址计算，非数据值）
- **MEM 阶段**：`mem_result = rd_val`（Trace 提供的内存加载真值）
- **WB 阶段**：`registers[rd] = mem_result`（写回加载值，非 alu_result）
- **转发网络**：Load 在 EX/MEM 阶段时 *不参与转发*（因为 `alu_result` 此时存的是地址，不是数据），必须等进入 MEM/WB 阶段后通过 `mem_result` 转发
- **Load-Use 停顿**：如果下一条指令在 ID 阶段检测到 EX/MEM 中的 Load 的 rd 是自己的 rs1 或 rs2，触发一个周期的停顿

**这是模拟器中最关键的指令类型**，因为它是唯一一种 `alu_result` 和实际数据不一致的情况，也是唯一触发流水线停顿的指令类型。

---

### 5. Store 指令（S 型——访存存储）

**示例指令：`sw a1, 0(a2)`**

Spike 日志中的表达：
```
core   0: 0x00000000800003a0 (0x00b62023) sw      a1, 0(a2)
core   0: 3 0x00000000800003a0 (0x00b62023) mem 0x0000000080001000 0x00000001
```

机器码 `0x00b62023`：
- opcode = `0x23` → OP_STORE
- rs1 = 12 (x12/a2) —— 基址寄存器
- rs2 = 11 (x11/a1) —— 待存储数据来源

关键差异：

| 字段 | 值 | 说明 |
|-----|-----|------|
| rs1 | 12 (x12) | 基址（参与依赖追踪） |
| rs2 | 11 (x11) | 数据源（参与依赖追踪） |
| **rd** | **0** | **S 型无 rd，强制清零** |
| mem_addr | 0x80001000 | 来自提交行的 `mem 0x...` |

Spike 的提交行对于 Store 指令输出格式为 `mem 地址 写入值`，因此 `parse_spike_log.py` 可以解析出 mem_addr。

流水线行为：
- **EX 阶段**：`alu_result = mem_addr`（存储地址计算）
- **MEM 阶段**：无特殊操作（锁存器透传）
- **WB 阶段**：`rd == 0`，不回写（x0 硬连线）
- **转发网络**：rs1 和 rs2 都参与依赖追踪，即 Store 指令可能因为需要读取 rs1/rs2 的值而触发转发，也可能因为后面的指令依赖 Store 的 rs1/rs2 而受益于转发（但 Store 本身不产生结果，只能作为转发的消费者）

---

### 6. Branch 指令（B 型——条件分支）

**示例指令：`beq a1, a2, label`**

Spike 日志中的表达：
```
core   0: 0x0000000080000040 (0x00c58863) beq     a1, a2, 16
core   0: 3 0x0000000080000040 (0x00c58863)
```

| 字段 | 值 | 说明 |
|-----|-----|------|
| rs1 | 11 (x11/a1) | 比较源1（参与依赖追踪） |
| rs2 | 12 (x12/a2) | 比较源2（参与依赖追踪） |
| **rd** | **0** | **B 型无 rd，强制清零** |
| mem_addr | 0 | 无访存 |
| rd_val | 0 | 提交行无寄存器写回值 |

注意 Spike 的提交行在 Branch 指令后只有 PC 和机器码，没有 x 寄存器写回信息——因为 Branch 不写寄存器。但 Spike 可能会（取决于日志级别）额外输出分支是否跳转的信息，不过 `parse_spike_log.py` 的 commit 正则要求 `x` 或 `mem` 关键字，对于 Branch 的提交行很可能匹配不到任何有效信息，因此 `rd_val` 和 `mem_addr` 都保持默认值 0。

流水线行为：
- **EX 阶段**：`alu_result = rd_val = 0`（值无意义）
- **WB 阶段**：`rd == 0`，不回写
- **转发网络**：rs1/rs2 参与依赖追踪

⚠ **当前模拟器未实现分支预测**。Branch 指令仅像其他指令一样流过流水线、增加指令计数，但不改变 PC。由于 Trace 中的指令顺序已经是动态执行顺序（分支结果已经在 Spike 中解析完毕），模拟器只需要按顺序消费 Trace 队列即可，不需要自己处理分支方向。

---

### 7. Jump 指令（J 型 JAL 和 I 型 JALR）

**示例：`jal ra, func`**

Spike 日志中的表达：
```
core   0: 0x0000000080000020 (0x008000ef) jal     ra, 8
core   0: 3 0x0000000080000020 (0x008000ef) x1 0x0000000080000024
```

JAL 的机器码 `0x008000ef`：
- opcode = `0x6F` → OP_JUMP
- rd = 1 (x1/ra)

| 字段 | JAL (`0x6F`) | JALR (`0x67`) |
|-----|--------------|---------------|
| rs1 | **0**（强制清零，J 型无 rs1） | 从 bit[19:15] 提取 |
| rs2 | **0**（强制清零） | **0**（I 型无 rs2） |
| rd | 正常提取 | 正常提取 |
| rd_val | 提交行的 x 值 = **pc+4** | 同 JAL |

提交行中 `x1 0x0000000080000024` 即返回地址（pc+4），这正是 Trace 记录中 `rd_val` 的值。

流水线行为：
- **EX 阶段**：`alu_result = pc + 4`（返回地址，由模拟器自行计算，**不使用 `rd_val`**）
- **WB 阶段**：`registers[rd] = alu_result = pc + 4`
- **转发网络**：JAL 无源寄存器依赖（rs1=rs2=0），JALR 依赖 rs1

⚠ 当前模拟器**不刷新流水线**。Jump 指令之后的指令在 Trace 队列中已经是跳转目标处的指令，因此模拟器只是顺序消费。在真实硬件中 Jump 后的指令会被冲刷，但在 Trace 驱动模式下不需要——Trace 已经记录了正确的程序顺序。

---

### 8. System 指令（I 型——系统调用）

**示例指令：`ecall`**

Spike 日志中的表达：
```
core   0: 0x0000000080000044 (0x00000073) ecall
```

机器码 `0x00000073`：
- opcode = `0x73` → OP_SYSTEM
- rs1 = 0、rd = 0（ecall 不使用寄存器字段）

System 指令在 Spike 日志中通常没有提交行（如 ecall 触发 trap 后执行流改变），日志内容取决于具体指令。

流水线行为与 IntAlu 相同：`alu_result = rd_val`，WB 阶段写回 register。当前模拟器没有对 System 指令做特殊处理。

| 指令类别 | 编码格式 | OpType | rs1 | rs2 | rd | mem_addr | rd_val | 是否写回 reg |
|---------|---------|--------|-----|-----|----|---------|--------|------------|
| add (R) | R | IntAlu | ✔ | ✔ | ✔ | 0 | 计算结果 | ✔ |
| addi (I) | I | IntAlu | ✔ | ✗(0) | ✔ | 0 | 计算结果 | ✔ |
| lui (U) | U | IntAlu | ✗(0) | ✗(0) | ✔ | 0 | 计算结果 | ✔ |
| lw (I) | I | Load | ✔(基址) | ✗(0) | ✔ | Spike日志不输出 | 加载值 | ✔(mem_result) |
| sw (S) | S | Store | ✔(基址) | ✔(数据) | ✗(0) | ✔(地址) | 无关 | ✗ |
| beq (B) | B | Branch | ✔ | ✔ | ✗(0) | 0 | 0 | ✗ |
| jal (J) | J | Jump | ✗(0) | ✗(0) | ✔ | 0 | pc+4 | ✔(pc+4) |
| jalr (I) | I | Jump | ✔ | ✗(0) | ✔ | 0 | pc+4 | ✔(pc+4) |
| ecall (I) | I | System | 0 | 0 | 0 | 0 | 0 | ✗ |

## 五、流水线视角下的三类指令行为

从模拟器流水线的角度看，六种 OpType 可以归纳为**三种行为模式**：

### 第一类：标准计算型（IntAlu / System）
- EX 阶段 `alu_result = rd_val`（直接复用 Trace 中的真值）
- WB 阶段 `registers[rd] = alu_result`
- 这是模拟器中最简单的类型，Trace 提供了完整的结果，模拟器只需按序写回

### 第二类：访存型（Load / Store）
- **Load** 的两阶段写回机制：EX 阶段地址 → MEM 阶段取数 → WB 阶段写回；转发网络特殊处理；可能触发 Load-Use 停顿
- **Store** 的地址计算：EX 阶段计算地址，但无写回；其对 rs1/rs2 的依赖仍参与转发网络

### 第三类：控制流型（Branch / Jump）
- **Branch** 在 Trace 驱动模式下不需要分支预测——Trace 顺序已隐含正确的动态执行路径；不作为控制流指令处理
- **Jump** 写回 pc+4 作为返回地址；不触发流水线冲刷

这种分类也指明了未来改进的方向：
- 访存延迟建模需要让 Load 的 MEM 阶段产生停顿
- 分支预测需要在模拟器中引入预测逻辑和预测错误时的流水线冲刷
- 多发射需要让 ID 阶段一次处理多条指令并处理更复杂的数据依赖

## 六、总结

重新审视指令 Trace 的结构后，可以清晰地看到：

1. **Trace 记录了架构可见状态**（PC、寄存器索引、访存地址、计算结果），但不记录架构不可见的流水线状态（旁路、停顿、冲刷）。这正是 Trace 驱动的核心思想——外部提供"正确答案"，模拟器只建模"到达答案的延迟"。

2. **rs1/rs2 的强制清零规则**是 Trace 正确性的基石。I 型指令的 rs2、U 型指令的 rs1/rs2、S/B 型的 rd、JAL 的 rs1/rs2 都必须强制清零。如果不清零，转发网络会产生错误的依赖关系，导致错误的操作数值传递，最终模拟出错误的性能行为。

3. **rd_val 承载的语义因指令类型而异**——对 IntAlu 是计算结果，对 Load 是内存加载值，对 Jump 是返回地址（pc+4），对 Store/Branch 则无意义。模拟器必须根据 op_type 区分使用方式。

4. **Spike 日志格式的局限性**：Load 指令的提交行不输出 `mem` 信息（只输出 rd 的寄存器写回值），因此 `mem_addr` 字段在 Load 指令中为 0。Load 的访存地址由 `rs1 + 符号扩展立即数` 构成，要完整解析需要脚本维护寄存器状态表追踪 `rs1` 的值。这是一个已知的可改进点，未来可以通过增强 `parse_spike_log.py` 来解决，为精确建模缓存行为提供必要的访存地址信息。
