## 数据冒险

RAW：Read After Write，读后写冒险。指的是指令1写寄存器，指令2读寄存器，指令2在指令1之前执行了，所以指令2读到的寄存器值是错误的。
WAR：Write After Read，写后读冒险。指的是指令1读寄存器，指令2写寄存器，指令2在指令1之前执行了，所以指令1读到的寄存器值是错误的。
WAW：Write After Write，写后写冒险。指的是指令1写寄存器，指令2写寄存器，指令2在指令1之前执行了，所以指令2写入的值会覆盖指令1写入的值。

这其中，只有一种冒险会造成处理器的停顿，那就是load use冒险，这是一种内存中取出数据立刻就要在ex阶段使用的RAW冒险。其他的冒险都可以通过转发来解决。

## 在模拟器中实现转发和停顿

刚刚提到，除了load use之外，其他的冒险都可以通过转发解决，所以我们先来看一看转发如何在模拟器当中实现。

### 转发的本质

转发（Forwarding，也称 Bypassing）的核心思想是：**不等结果写回寄存器文件，而是将流水线中已经计算好的结果直接"短路"给当前指令使用**。

在经典五级流水线中，一个指令的结果要经过 EX → MEM → WB 三个阶段才会写回寄存器文件。而紧随其后的指令在 ID 阶段就需要这个值。如果不做转发，就需要插入停顿等待写回——转发就是在 ID 阶段读取寄存器时，检测到流水线后面已经有计算好的结果，直接拿来用。

### 转发网络的实现

转发网络实现在 `forward()` 方法中（`simulator.rs:226`），分为三个优先级：

```
优先级 1: EX/MEM 锁存器 ← 最近一条指令的 ALU 结果
优先级 2: MEM/WB 锁存器 ← Load 的取数结果或其他指令的 ALU 结果  
优先级 3: 寄存器文件 ← 默认值
```

关键代码逻辑如下：

```rust
fn forward(&mut self, rs: u8) -> u64 {
    if rs == 0 {
        return 0; // x0 硬连线为 0，永不转发
    }

    // 优先级 1：EX/MEM 阶段
    if let Some(ref em) = self.em_latch {
        if em.record.rd == rs && em.record.op_type != OpType::Load {
            self.forward_count += 1;
            return em.alu_result;
        }
    }

    // 优先级 2：MEM/WB 阶段  
    if let Some(ref mw) = self.mw_latch {
        if mw.record.rd == rs {
            self.forward_count += 1;
            return if mw.record.op_type == OpType::Load {
                mw.mem_result
            } else {
                mw.alu_result
            };
        }
    }

    // 优先级 3：寄存器文件
    self.registers[rs as usize]
}
```

#### 优先级设计

转发优先级的设计遵循 **"时序上最近的结果"** 这一原则：

| 优先级 | 来源 | 时序位置 |
|--------|------|----------|
| 1 | EX/MEM（em_latch） | 上一条指令刚算出的结果 |
| 2 | MEM/WB（mw_latch） | 上两条指令的结果或 Load 取数 |
| 3 | 寄存器文件 | 已提交的体系结构状态 |

指令在 EX 阶段计算的结果写入 `alu_result`，这个值在下一个周期进入 EX/MEM 锁存器。此时如果下一条指令已经进入 ID 阶段，并且源寄存器号匹配，就可以从 EX/MEM 直接转发——这覆盖了最常见的"相邻指令间 RAW 冒险"。

如果两条指令间还隔了一条独立指令，EX/MEM 锁存器中已经不是目标值了，需要从 MEM/WB 锁存器中获取。

### Load 指令的特殊处理

Load 指令在流水线中的行为与其他指令不同：

1. **EX 阶段**计算的是**有效地址**，写入 `alu_result`
2. **MEM 阶段**才从内存取回真正的数据，写入 `mem_result`

所以当 Load 在 EX/MEM 锁存器中时，`alu_result` 存放的是地址而非数据，**不能用于转发**。这也是转发优先级 1 中排除 `Load` 的原因：

```rust
if em.record.rd == rs && em.record.op_type != OpType::Load {
    self.forward_count += 1;
    return em.alu_result;
}
```

而当 Load 推进到 MEM/WB 锁存器时，`mem_result` 已经包含正确的加载值，可以从优先级 2 正常转发：

```rust
if mw.record.rd == rs {
    self.forward_count += 1;
    return if mw.record.op_type == OpType::Load {
        mw.mem_result     // Load 的取数结果
    } else {
        mw.alu_result     // 其他指令的 ALU 结果
    };
}
```

### EX 阶段的 ALU 结果计算

为什么不同指令类型的 `alu_result` 含义不同？这是由各种指令在 EX 阶段的实际行为决定的：

```rust
fn stage_ex(&mut self) {
    let mut em = self.de_latch.take();
    if let Some(ref mut latch) = em {
        match latch.record.op_type {
            OpType::Load | OpType::Store => {
                // 访存指令：ALU 计算有效地址（来自 Trace 的 mem_addr）
                latch.alu_result = latch.record.mem_addr;
            }
            OpType::Jump => {
                // JAL / JALR：返回地址 = pc + 4，写回 rd
                latch.alu_result = latch.record.pc + 4;
            }
            _ => {
                // IntAlu / Branch / System：来自 Trace 的真值
                latch.alu_result = latch.record.rd_val;
            }
        }
    }
    self.em_latch = em;
}
```

| 指令类型 | alu_result 含义 | 用途 |
|---------|----------------|------|
| Load/Store | 访存地址 (mem_addr) | MEM 阶段访存使用 |
| Jump | 返回地址 (pc+4) | 写回 rd（如 JAL 的链接） |
| IntAlu/Branch/System | 运算结果 (rd_val) | 写回 rd 或供转发 |

区分这些类型的意义在于：**转发网络依赖 `alu_result` 作为数据来源**，必须确保转发的值在语义上是正确的。例如，如果误将 Load 的地址当作数据转发，就会产生错误结果。

### Load-Use 冒险检测与停顿

当 Load 指令的结果被紧邻的下一条指令使用时，即使转发也无法解决——因为 Load 在 EX 阶段只有地址，要等到 MEM 阶段结束才有数据。这时唯一的选择是**停顿一个周期**。

冒险检测在 ID 阶段进行（`simulator.rs:174-184`）：

```rust
// 检测条件：EX/MEM 中有 Load，且 FD 中的指令依赖其 rd
if let Some(ref em) = self.em_latch {
    if em.record.op_type == OpType::Load && em.record.rd != 0 {
        if let Some(ref fd) = self.fd_latch {
            if fd.record.rs1 == em.record.rd || fd.record.rs2 == em.record.rd {
                hazard = true;
            }
        }
    }
}
```

检测条件同时满足时判定为 Load-Use 冒险：
1. **EX/MEM 锁存器中有一条 Load 指令**（尚未取回数据）
2. **IF/ID 锁存器中有一条指令**，其任一源寄存器与该 Load 的目标寄存器相同

停顿的处理包括三个动作：

```rust
if hazard {
    self.stall = true;      // 设置停顿标志
    self.stall_count += 1;  // 统计停顿次数
    self.de_latch = None;   // ID 阶段插入气泡（NOP）
    return;
}
```

- **stall = true**：通知 IF 阶段冻结 fd_latch（stall 为 true 时 IF 不取新指令）
- **de_latch = None**：向 ID/EX 锁存器写入空值（插入气泡），相当于硬件中的流水线 stall 信号
- 下个周期 `stall` 被清除后，之前冻结的 fd_latch 正常推进到 ID 阶段，而此时 Load 已经进入 MEM/WB，可通过转发获取数据

#### 时序图解

以 `lw x1, 0(x4)` 后紧跟 `add x2, x1, x0` 为例：

```
周期  |  IF   |  ID   |  EX   |  MEM  |  WB
──────┼───────┼───────┼───────┼───────┼───────
  T   |  add  |  lw   |       |       |       ← 检测到冒险
 T+1  | [冻结]| [气泡]|  lw   |       |       ← 停顿一周期
 T+2  |  nop  |  add  | [气泡]|  lw   |       
 T+3  |       |  nop  |  add  | [气泡]|  lw   ← add 通过转发拿到 lw 结果
```

在 T+3 周期，add 进入 EX 阶段，lw 在 MEM/WB 锁存器中已有 `mem_result`，通过转发网络的优先级 2 正确取得数据。

### 转发生效场景分析

不同指令间隔下的转发行为：

| 场景 | 指令序列 | 转发来源 | 是否需要停顿 |
|------|---------|----------|-------------|
| 相邻 ALU→ALU | add x1; add x2, x1 | EX/MEM | 否 |
| 跨一条 ALU→ALU | add x1; nop; add x2, x1 | MEM/WB | 否 |
| 相邻 Load→ALU | lw x1; add x2, x1 | — | **是，1 周期** |
| 跨一条 Load→ALU | lw x1; nop; add x2, x1 | MEM/WB | 否 |
| 连续两条 Load | lw x1; lw x2; add x3, x1, x2 | MEM/WB + 停顿 | 部分（仅依赖第二条 Load 时） |
