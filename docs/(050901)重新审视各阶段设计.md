# 重新审视各阶段设计

同样和上一篇一样的理由，由于过度使用AI，我们丢失了对于模拟器本身结构的理解。这在堆进度的时候无可厚非，但在我们以学习原理、实践原理为目的的情况下，这就是一个错误了。

在本篇文章中，我们将会阅读代码，结合上一章的指令结构的内容，重新审视各个结构的设计，看看它们是如何实现的，以及它们之间的关系。

## 从哪里开始？

我们还是从main.rs开始。

```rust
/// 主启动函数：解析输入 Trace 文件并将其加载至流水线模拟器，
/// 以周期驱动方式推进流水线直至全部排空
fn main() -> io::Result<()> {}
```

现在main.rs中只有一个main函数，作用是解析输入的Trace文件并加载。那么这里就有第一个问题，加载后的结构是什么，怎么传给模拟器的，这就是我们第一个分界点。

```rust
// ── 第一步：将全部 Trace 记录读入内存 ──\\
let raw: TraceRecordRaw =
    unsafe { std::ptr::read_unaligned(buffer.as_ptr() as *const _) };
records.push(raw.into_record());
```

所以说这个衔接了Trace解析和模拟器的变量是records。我们再看它是怎么到模拟器并被执行的。

```rust
// ── 第二步：加载至模拟器并按周期推进流水线 ──
let mut sim = Simulator::new();
sim.load_trace(records);

println!("[*] 开始按流水线周期推演...");
while !sim.is_done() {
    sim.step_cycle();
}

println!("[*] 流水线推演完毕。");
sim.print_statistics();
```

这样依赖我们就知道关键在哪里了，关键就在于Simulator的step_cycle方法。

## Simulator是如何执行Trace的？

当然了，我们首先要去定位到step_cycle方法的位置。

这是在simulator/core.rs中，实际上着就是把五个阶段的stage方法全部执行一遍，然后cycle+1，当然，这就意味着这五个阶段是在同一个周期中完成的。并且执行的顺序是wb到if，这是为了避免数据被覆盖吧。后面的先把上一周期的生产的数据消费了，然后才好生产新的数据。

```rust
self.stage_wb();
self.stage_mem();
self.stage_ex();
self.stage_id();
self.stage_if();
self.cycle += 1;
```

那么我们暂时先不进入具体的流水线阶段，我们继续看这个文件的其他部分。注意到了is_done方法，这也在main中出现过。我们看看是如何评价的。是通过trace为空并且各个阶段的latch也为空评价的。很显然。latch在这里相当于是流水线寄存器，起到了保存上一周期产生的状态的作用。

那么我们就得看看latch的构成了。

`struct Latch`是在mod.rs中定义的，这个文件也定义了`struct Simulator`。所以可以注意到Latch是Simulator的成员，这是很自然的。可以想到如果我们之后要做多发射改造，各个寄存器也该放到这里。

然后看到Latch的具体内容，实际上就是单纯的字段打包而已，说实话，命名为Latch我是觉得不太好的，因为锁存器是之前学用RTL设计的时候尽量避免的。与其叫做latch不如直接叫PipelineRegister之类的名字，不过现阶段就不改了。改多发射的时候再来修改这个东西。

实际上代码读到这一步，接下来是什么情况就已经比较清晰了，整个模拟器的全貌也基本展现在了我们的面前。但我们现在依然要走完最后一步。进入到`simulator/pipeline.rs`中，看看五个阶段是怎么实现的。

## 五个阶段分别是怎么实现的？

我们将要关注这五个阶段的实现，关键点到底是哪里。
毫无疑问，关键点绝对是hazard和forward的实现。
所以我们先看看主体的各个阶段各自执行的实现。

对于wb，就是将load到的数据或者计算得到的数据协会寄存器堆。

对于mem，进行load和store，但是store在这种trace模拟器中就不用了；不过这里有一个问题，按照正常的道理，这个数据应该是在wb阶段才能从mem中吐出来的，现在直接在mem阶段拿到，我们得理一理是否合理。这里是构建的mw的这个latch，那么这个latch得到的结果实际上这里就是最后的值了，所以我认为这里在时序语义上依然是有问题的，但是对于执行来说却没有问题。因为我们不需要对load的数据进行各种处理，所以整个过程就简化了很多了。但是从打拍来说，依然应该是在wb阶段才拿到数据的，但是这就要求存在一个额外的独立于pipeline的LSU，这似乎是不符合我们现在的设计方案的，到底需不需要一个独立的LSU甚至ALU呢？

需要的，我觉得是需要的。如果之后我们要想更进一步的超标量上面来做工夫，那么就必须要有各个独立的执行单元。那么现在这种纯纯的阶段实现就不具备足够的扩展性了。这是一个待做的点！

那么包括hazard、forward，这些也无疑是需要单独的模块来做的。直接用阶段实现，在效果上成立，但是不符合我们的建模语义。所以是绝对不行的。

那么我将会中止这一篇文章的撰写，转而准备在下一篇文章进行重构了。

之前虽然重构过一次，但是缺乏人工干预，导致重构的结果并不理想，这是一个很关键的教训。
