# TraceRecord 能力边界与各模块设计关联分析

> 回答 `docs/(051701)重写后代码满足需求了吗.md` 中提出的核心问题：
> **TraceRecord 能够做到什么、不能做到什么，以及各模块的设计是如何关联（利用或补偿）这些能力边界的。**

---

## 一、TraceRecord 的能力全景

### 1.1 字段清单（32 字节 / 条指令）

| 字段 | 类型 | 何时有效 | 语义 |
|------|------|---------|------|
| `pc` | u64 | 始终 | 指令地址 |
| `inst` | u32 | 始终 | 原始 32 位机器码（可用于重解码） |
| `rs1` | u8 | 有源操作数时 | 源寄存器 1 索引（0 = x0 或无） |
| `rs2` | u8 | 有源操作数时 | 源寄存器 2 索引（0 = x0 或无） |
| `rd` | u8 | 有写回时 | 目标寄存器索引（0 = x0 或不写回） |
| `op_type` | OpType | 始终 | 指令大类：IntAlu / Load / Store / Branch / Jump / System |
| `mem_addr` | u64 | Load/Store 时 | 有效访存地址（当前仅 Store 的地址被消费） |
| `rd_val` | u64 | 写寄存器时 | **预计算的真值**（IntelAlu 的结果 / Load 的加载值 / Jump 的 pc+4） |

### 1.2 能力划分——TraceRecord 提供什么

| 能力 | 对应字段 | 说明 |
|------|---------|------|
| ✅ 寄存器数据依赖（RAW） | `rs1`, `rs2`, `rd`, `op_type` | 据此检测 Load-Use 冒险、驱动转发网络 |
| ✅ 最终计算结果 | `rd_val` | 模拟器无需真实 ALU 计算，直接使用 Trace 真值 |
| ✅ 指令类型分类 | `op_type` | 决定流水线各阶段的行为差异（地址计算 / 写回选择） |
| ✅ 访存有效地址（Store） | `mem_addr` | 建模 Store 指令的地址计算 |
| ✅ 加载值（Load） | `rd_val` | Load 指令的 `rd_val` **就是**从内存加载的值 |
| ✅ 跳转返回地址（Jump） | `pc` | 模拟器自行计算 `pc + 4` 作为 JAL 的链接地址 |
| ✅ 动态执行顺序 | Trace 顺序 | Trace 已记录 Spike 真实执行路径，模拟器顺序消费即可 |

### 1.3 能力边界——TraceRecord 不提供什么

| 缺失信息 | 后果 | 设计影响 |
|---------|------|---------|
| ❌ **源寄存器值**（rs1_val, rs2_val） | ALU 无法真实计算 | **补偿**：转发网络 + 寄存器文件从锁存器重建 |
| ❌ **分支方向 / 目标地址** | 无法建模分支预测 | **简化**：Branch 流经流水线但不改变控制流，不触发冲刷 |
| ❌ **Cache 命中/缺失信息** | 无法建模访存延迟变化 | **硬编码**：MEM 阶段固定 1 周期，LSU 对象存在但未接入 |
| ❌ **多周期执行延迟** | 无法区分不同 ALU 指令的延迟 | **未实现**：ExecUnit 类存在但未接入流水线 |
| ❌ **异常 / 中断标志** | 无法建模异常处理 | **不相关**：性能建模不需要异常语义 |
| ❌ **立即数** | 需要从 inst 重解码 | **不需要**：Trace 已提供 `rd_val`，ALu 无需立即数 |
| ❌ **Store 数据值** | 不知道写入内存的具体值 | **不需要**：Trace 驱动模拟器不建模内存状态 |
| ❌ **指令 ID / ROB 编号** | 无法建模乱序执行 | **未实现**：当前仅顺序流水线 |
| ❌ **多线程同步事件** | 无法建模多核 | **不相关**：当前仅单核（论文指出单核纯 Trace 可行） |

## 二、各模块与 TraceRecord 能力边界的关联

### 模块依赖总图

```
                  ┌──────────────┐
                  │  TraceRecord  │  ← 32 字节二进制记录
                  │   (trace.h)   │
                  └──────┬───────┘
                         │ 被所有模块依赖
    ┌────────┬───────────┼───────────┬──────────┐
    ▼        ▼           ▼           ▼          ▼
  latches   alu        hazard     simulator   lsu/exec_unit
  (数据载体) (结果路由)  (冒险检测)  (流水线编排) (时序模型-未接入)
```

---

### 2.1 Trace 解析层 (`trace.h`, `trace.cc`)

**模块职责**：将二进制 Trace 文件反序列化为 `TraceRecord`，并将 `OpType` 从原始 u8 映射为枚举。

**利用的 Trace 能力**：
- 二进制格式 (`TraceRecordRaw`) 32 字节小端序，`__attribute__((packed))` 保证内存布局与磁盘一致
- `op_type_from_u8()` 完成 `u8 → OpType` 的安全映射（非法值 → `Unknown`）

**受限于 Trace 缺失**：
- **无语义验证**：解析层不校验 `rd=0` 是否与指令类型一致（如 Store/Branch 的 `rd` 应为 0），这些由 Trace 生成端（`parse_spike_log.py`）保证
- **无完整性检查**：不校验 `rs1`/`rs2` 的合法范围（应在 0–31）
- **无格式头**：纯连续记录，无魔数、无计数头、无校验和——文件损坏会静默产生错误结果

**设计评价**：✅ 足够且高效。解析层的职责单一——转换格式，不涉及语义。

---

### 2.2 流水线锁存器 (`latches.h`)

**模块职责**：定义四个级间锁存器和不可变快照 `PipelineSnapshot`，作为流水线状态的数据载体。

**四个锁存器的信息渐变**：

| 锁存器 | 携带数据 | 新增的运行时值 |
|--------|---------|-------------|
| `IfIdLatch` | `TraceRecord` | （无，仅 Trace 静态信息） |
| `IdExLatch` | `TraceRecord` + `rs1_val` + `rs2_val` | 经转发网络解析的源操作数 |
| `ExMemLatch` | `TraceRecord` + `alu_result` | ALU 计算结果（地址或 `rd_val`） |
| `MemWbLatch` | `TraceRecord` + `alu_result` + `mem_result` | 访存结果（仅 Load 有效） |

**利用的 Trace 能力**：
- 锁存器从始至终携带 `TraceRecord`，使得任意阶段都可查询指令元信息（PC、寄存器号、op_type）
- `PipelineSnapshot` 捕获周期开始时的完整流水线状态，作为组合逻辑的不可变输入

**补偿的 Trace 缺失**：
- Trace 不提供源寄存器值 (`rs1_val`/`rs2_val`) → `IdExLatch` 在 ID 阶段通过 `forward_value()` 动态填充
- Trace 不提供 ALU 结果（对 Load/Store 而言） → `ExMemLatch` 通过 `Alu::compute()` 动态计算
- 锁存器使用 `std::optional` —— 空值表示气泡（NOP），这是硬件流水线寄存器的软件对应

**设计评价**：✅ 结构清晰。`PipelineSnapshot` 设计是实现"组合逻辑与时序逻辑分离"的关键。

---

### 2.3 组合 ALU (`alu.h`, `alu.cc`)

**模块职责**：纯组合函数，根据 `OpType` 将正确的值路由到 `alu_result`。

**实现逻辑**：

```cpp
static uint64_t compute(const TraceRecord& record, uint64_t rs1_val, uint64_t rs2_val) {
    switch (record.op_type) {
        case OpType::Load:
        case OpType::Store:     return record.mem_addr;   // 有效地址
        case OpType::Jump:      return record.pc + 4;     // 返回地址
        default:                return record.rd_val;     // 预计算结果
    }
}
```

**利用的 Trace 能力**：
- `rd_val` → IntAlu / Branch / System 的直接结果（**不需要真 ALU 计算**）
- `mem_addr` → Load / Store 的有效地址（**不需要解码立即数 + 加法**）
- `pc` → JAL 返回地址计算（`pc + 4`）

**受限于 Trace 缺失**：
- **`rs1_val` 和 `rs2_val` 完全不使用**（参数签名保留但注释掉）——因为 Trace 已提供 `rd_val`
- **无法区分子类型**——所有 IntAlu 指令（add、sub、shift、multiply）统一返回 `rd_val`，假设执行延迟相同
- **Load 的 `alu_result` (= `mem_addr`) 是死数据路径**——当前流水线中从不被消费：
  - EX/MEM 转发显式排除 Load（`op_type != OpType::Load`）
  - MEM/WB 转发对 Load 使用 `mem_result`（= `rd_val`），非 `alu_result`
  - WB 写回对 Load 使用 `mem_result`

> ⚠ **死数据路径结论**：Load 的 `mem_addr` 为 0 也**不影响当前模拟器正确性**。当需要建模 Cache 行为时，这个路径才需要激活。

**设计评价**：✅ 作为 mux 是正确的。`rs1_val`/`rs2_val` 参数保留为未来真实 ALU 计算预留接口。

---

### 2.4 冒险检测单元 (`hazard.h`, `hazard.cc`)

**模块职责**：纯组合函数，检测 Load-Use 冒险并解析转发来源。

**利用的 Trace 能力**：
- `rs1` / `rs2` / `rd` → 寄存器依赖匹配
- `op_type == Load` → 区分 Load（数据在 MEM 之后才就绪）与非 Load（EX 阶段即有结果）

**检测的冒险类型**：

| 冒险类型 | 是否检测 | 检测条件 | 处理方式 |
|---------|---------|---------|---------|
| Load-Use RAW | ✅ 检测 | `de.record.op_type == Load` 且 `de.record.rd != 0` 且 `fd.record.rs1/rs2 == de.record.rd` | 停顿 1 周期（`stall_if = stall_id = true`） |
| ALU→ALU RAW | ✅ 不检测（直接转发） | — | 转发网络自动解决 |
| 分支预测错误 | ❌ 不检测 | 无分支元数据 | — |
| 内存冒险 | ❌ 不检测 | 无内存消歧信息 | — |

**关键设计细节——两个转发解析器的不同角色**：

代码中存在两个逻辑完全相同但**运行时机不同**的转发解析器：

| 属性 | `HazardUnit::resolve_forward()` | `Simulator::forward_value()` |
|------|-------------------------------|------------------------------|
| 运行时机 | 组合逻辑阶段（快照） | 时序提交阶段（ID 提交时） |
| 状态视图 | **周期开始时的快照**（不可变） | **实时锁存器**（EX/MEM 已更新） |
| 结果用途 | **仅用于单元测试**（`fwd_src1`/`fwd_src2` 在运行时不被消费） | **实际操作数解析**（写入 `IdExLatch.rs1_val/rs2_val`） |
| 同周期 EX→ID | ❌ 看不到（生产指令还在 snapshot 的 DE 中） | ✅ 能看到（生产指令已从 DE 移到 EM） |

> ⚠ **重要**：`HazardResult::fwd_src1` 和 `fwd_src2` 在 `HazardUnit::resolve()` 中被计算，但在 `simulator.cc` 的提交阶段**从未被读取**。它们是运行时死代码，仅被单元测试 (`hazard_test.cc`) 验证。**实际生效的转发解析器只有一个**：`Simulator::forward_value()`。

**转发优先级逻辑（两者一致）**：

```
优先级 1: EX/MEM 锁存器 — rd 匹配 且 op_type != Load → alu_result
优先级 2: MEM/WB 锁存器 — rd 匹配 → Load 用 mem_result，否则用 alu_result
优先级 3: 寄存器文件 — 默认值
特殊规则: rs == 0 → 始终返回 0（x0 硬连线）
```

**受限于 Trace 缺失**：
- **无分支方向** → `HazardResult` 的 `flush_if` / `flush_id` 字段**从未被设置为 `true`**（预留的冲刷接口未激活）
- **无多周期延迟** → 仅检测 Load 这一种需要停顿的 RAW 冒险（其他 RAW 全由转发解决）

**设计评价**：✅ 核心逻辑正确。⚠ `resolve_forward` 的转发结果在运行时未被消费——存在逻辑重复的维护风险（若一个函数的优先级规则被修改而另一个未同步，单元测试可能通过但运行时行为不正确）。

### 2.5 模拟器核心 (`simulator.h`, `simulator.cc`)

**模块职责**：拥有全部流水线状态，编排三阶段周期推进，实现操作数转发网络，输出性能统计。

#### 三阶段周期推进（`step_cycle()`）

```
Phase 1 — 快照:     PipelineSnapshot snap = snapshot();
                    → 捕获周期开始时的不可变状态

Phase 2 — 组合逻辑:  HazardResult hazard = HazardUnit::resolve(snap);
                    → 纯函数，产生控制信号（停顿/转发来源）

Phase 3 — 时序逻辑:  反向顺序提交（WB → MEM → EX → ID → IF）
                    → 实现同周期 EX→ID 转发
```

**反向提交的必要性**：如果 ID 先于 EX 执行，ID 阶段读取的 EM 锁存器仍是上一周期的值。反向顺序确保 ID 读取时 EX 已将新结果写入 EM 锁存器，从而 **Load 的 `mem_result` 能在同一周期经 MWB 转发给消费者**（对于跨一条指令的 Load-Use，MEM 阶段先更新 MW 锁存器，随后 ID 阶段的 `forward_value()` 就能从中取到最新值）。

#### 转发网络（`forward_value()`）

```cpp
uint64_t forward_value(uint8_t rs) {
    if (rs == 0) return 0;                              // x0 硬连线
    if (em_latch_.rd == rs && em_latch_.op != Load)     // 优先级 1：EX/MEM
        { forward_count++; return em_latch_.alu_result; }
    if (mw_latch_.rd == rs) {                           // 优先级 2：MEM/WB
        forward_count++;
        return (mw_latch_.op == Load) ? mw_latch_.mem_result : mw_latch_.alu_result;
    }
    return registers_[rs];                              // 优先级 3：寄存器文件
}
```

> 这是**实际生效的唯一转发解析器**。运行在 ID 提交阶段，读取的是已更新的实时锁存器状态。

#### 利用的 Trace 能力

| 流水线阶段 | 消费的 Trace 字段 | 作用 |
|-----------|-----------------|------|
| IF | `TraceRecord` 整体 | 从 Trace 队列入队到 `fd_latch_` |
| ID | `rs1`, `rs2`, `rd`, `op_type` | 解析操作数，触发转发 |
| EX | `op_type`, `mem_addr`, `rd_val`, `pc` | `Alu::compute()` 路由正确的值 |
| MEM | `rd_val` (Load 时) | `mem_result = rd_val`（直接使用 Trace 提供的加载值） |
| WB | `rd`, `op_type` | 决定写回 `mem_result` (Load) 还是 `alu_result` (其他) |

#### 补偿的 Trace 缺失

- **无源寄存器值** → 转发网络从锁存器动态重建
- **无 Cache 延迟** → MEM 阶段硬编码 1 周期（`mem_result` 直接从 Trace `rd_val` 取值）
- **无分支方向** → Branch 自然流经流水线，不触发冲刷
- **无多周期延迟** → 所有指令 EX 阶段统一 1 周期

#### 统计输出

```
周期数  │ IPC  │ 转发命中数 │ Load-Use 停顿数 │ 指令混合分布
```

- IPC = `inst_count` / `cycle`
- `forward_count`：在 `forward_value()` 中每次命中递增
- `stall_count`：每次 Load-Use 冒险触发停顿时递增
- `op_stats`：每条指令在 WB 阶段按其 `op_type` 分类计数

**设计评价**：✅ 核心编排正确。反向提交 + `forward_value` 精确建模了硬件转发时序。⚠ 三处未激活的扩展接口：
1. `ExecUnit` 对象不存在于 `Simulator` 中
2. `Lsu` 对象存在但完全未接入——`lsu_.execute()` / `lsu_.reserve()` 从未被调用
3. `HazardResult::flush_if` / `flush_id` 从未被设置

### 2.6 Load-Store Unit (`lsu.h`, `lsu.cc`)

**模块职责**：建模访存时序（延迟、流水化）。当前**完全未接入流水线**。

**实际状态**：

| 组件 | 状态 |
|------|------|
| `Lsu` 对象 | `Simulator` 成员变量 `lsu_` 在构造函数中初始化 |
| `lsu_.execute()` | **从未被调用**——MEM 阶段直接使用 `em.record.rd_val` |
| `lsu_.reserve()` | **从未被调用**——无结构冒险检测 |
| `lsu_.is_ready()` | **从未被调用**——无流水线停顿因 LSU 忙 |
| `lsu_.latency` | 默认值 1，无任何代码路径修改它 |
| `lsu_.pipelined` | 默认值 `true`，无任何代码路径修改它 |

MEM 阶段的**实际实现**（`simulator.cc:82-84`）：
```cpp
uint64_t mem_result = (em.record.op_type == OpType::Load)
    ? em.record.rd_val    // ← 直接从 Trace 取值，未经 LSU
    : 0;
```

**为什么这样设计是合理的**：
- `rd_val` 已经是 Spike 提供的**加载真值**——模拟器不需要"真的去访问内存"
- 当前只建模 L1 命中（1 周期）——MEM 阶段本身就是 1 个周期，不需要额外的定时器
- LSU 接口（`reserve` / `is_ready`）是为未来建模 cache miss（>1 周期）和结构冒险预留的

**未来若要激活 LSU**：
1. MEM 阶段需改为调用 `lsu_.reserve()` 获取数据就绪的周期号
2. 若数据未就绪 → 需停顿后续流水线阶段（新增停顿控制信号）
3. 需要 Trace 提供 cache miss 元数据（或由模拟器自行判断 miss）

**设计评价**：⚠ 存在但未接入。文档注释说是"建模访存时序"，但实际上完全绕过。对于当前目标（经典五级流水线，L1 命中）是正确的，但读者应清楚这是**预留接口**而非**有效模块**。

---

### 2.7 执行单元 (`exec_unit.h`, `exec_unit.cc`)

**模块职责**：定义可配置执行单元（名称、延迟、是否流水化）和工厂方法。**完全未接入流水线**。

**实际状态**：

| 组件 | 状态 |
|------|------|
| `ExecUnit` 对象 | **不存在于 `Simulator` 中**——无任何成员变量 |
| `is_ready()` | 从未被调用 |
| `reserve()` | 从未被调用 |
| `release()` | 从未被调用 |
| 工厂方法 | `default_int_alu()` / `default_lsu()` / `default_branch()` 仅在测试中使用 |

**该类的意图**（来自注释）：
- 乘法器：`latency=3, pipelined=true`
- 除法器：`latency=20, pipelined=false`（结构冒险）
- 每个 `OpType` 映射到一个 `ExecUnit`，EX 阶段根据单元状态决定是否停顿

**为什么当前不需要**：
- 所有指令在 Trace 驱动的模式下，EX 阶段统一 1 周期——`alu_result` 直接来自 Trace
- 没有真正的"执行"——`Alu::compute()` 只是路由值
- 在实现多周期 ALU 之前，`ExecUnit` 没有接入的必要

**设计评价**：✅ 前瞻性好的基础设施。明确标注为"未来扩展"，不会对读者产生误导。

---

### 2.8 主入口与统计 (`main.cc`)

**模块职责**：读取二进制 Trace 文件 → 加载到 `Simulator` → 循环 `step_cycle()` 直到 `is_done()` → 打印统计。

**利用的 Trace 能力**：
- 二进制格式的连续读取（`ifstream::read` 直接映射到 `TraceRecordRaw`）
- 顺序消费——Trace 顺序已隐含正确的动态执行路径

**受限于 Trace 缺失**：
- 无分支预测 → 不需要处理冲刷，不需要处理控制流改变
- 无异常 → 不需要处理 Trap

**未覆盖的测试路径**（来自测试覆盖率分析）：
- `TraceRecord::from_raw()` ——零测试覆盖
- 二进制文件部分损坏（最后一条记录不完整）——有代码处理但无测试
- 真实的 `test_c.trace` 集成测试——从未被自动化测试加载

**设计评价**：✅ 流程完整。⚠ 缺少对二进制解析路径的单元测试。

## 三、活代码 vs 死代码 vs 预留接口总览

## 四、Trace 能力边界 → 设计决策映射表

| Trace 提供的能力 | 哪个模块利用 | 如何利用 |
|----------------|------------|---------|
| `rs1` / `rs2` / `rd`（寄存器号） | HazardUnit, Simulator | 依赖检测 + 转发源匹配 |
| `rd_val`（预计算结果） | Alu, Simulator | 直接作为 ALU 结果 / Load 加载值 |
| `mem_addr`（Store 地址） | Alu | `alu_result = mem_addr`（Store 地址计算） |
| `op_type`（指令类型） | Alu, HazardUnit, Simulator | 决定路由逻辑、冒险检测条件、写回选择 |
| `pc`（指令地址） | Alu（Jump） | `alu_result = pc + 4`（返回地址） |
| Trace 顺序 | Simulator（IF 阶段） | 顺序消费 = 正确执行路径 |

| Trace 缺失的信息 | 影响哪个模块 | 补偿 / 简化策略 |
|----------------|------------|----------------|
| 源寄存器值 (`rs1_val`/`rs2_val`) | Alu, Simulator | 转发网络从锁存器动态重建 |
| 分支方向 / 目标 | HazardUnit, Simulator | Branch 作为 NOP 流经，不触发冲刷 → **高估 IPC** |
| Cache 命中/缺失 | LSU, MEM 阶段 | 硬编码 1 周期命中 → **低估停顿** |
| 多周期延迟 | ExecUnit, EX 阶段 | 统一 1 周期 → **不区分指令延迟** |
| 异常 / 中断标志 | Simulator | 不影响性能建模 |
| 立即数 | Alu | 不需要——`rd_val` 已提供结果 |
| 多核同步事件 | (全部) | 当前仅单核，论文指出单核纯 Trace 可行 |

## 五、回答：重写后代码满足需求了吗？

### 5.1 当前需求回顾

根据 README.md 和各历史文档，当前的需求是：

1. ✅ **经典五级流水线建模**（IF/ID/EX/MEM/WB）——**已满足**
2. ✅ **转发网络**（EX/MEM → MEM/WB → 寄存器文件三级优先级）——**已满足**
3. ✅ **Load-Use 冒险停顿**——**已满足**
4. ✅ **RISC-V 六种指令类型覆盖**（IntAlu / Load / Store / Branch / Jump / System）——**已满足**
5. ✅ **模块化拆分**（ALU、HazardUnit、LSU、ExecUnit、Latch）——**已满足**
6. ✅ **两相（三阶段）周期推进**（快照 → 组合逻辑 → 反向提交）——**已满足**
7. ✅ **30 个单元测试 + 集成测试**——**已满足**
8. ✅ **x0 硬连线**——**已满足**
9. ✅ **IPC 统计 + 指令混合分布统计**——**已满足**

### 5.2 当前未满足的需求（README "待做"）

| 待做项 | 当前状态 | 模块准备度 |
|--------|---------|----------|
| 分支预测模型 | 未实现。Branch 仅流经，`flush_*` 信号从未激活 | HazardUnit 有 `flush_if`/`flush_id` 字段预留 |
| 多发射 / 乱序执行 | 未实现 | Latch 需从 `optional` 改为数组 |
| 缓存缺失惩罚建模 | 未实现。LSU 完全未接入 | Lsu 类有 `latency`/`pipelined` 接口预留 |
| 统计可视化输出 | 未实现 | `print_statistics()` 是基础文本输出 |

### 5.3 架构健康度评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 核心功能完整性 | ⭐⭐⭐⭐⭐ | 经典五级流水线完整、转发网络正确、冒检测准确 |
| 模块化设计 | ⭐⭐⭐⭐⭐ | 七模块划分清晰，职责单一，依赖方向正确 |
| 测试覆盖 | ⭐⭐⭐⭐ | 30 个测试覆盖核心路径，但二进制解析路径和扩展接口未测 |
| 扩展性 | ⭐⭐⭐⭐ | `ExecUnit`、`Lsu`、`flush_*` 预留接口完整，接入成本低 |
| 死代码清理 | ⭐⭐⭐ | LSU 对象存在但未接入（内存中的僵尸对象）；`fwd_src*` 只用于测试 |
| 文档完整性 | ⭐⭐⭐ | 代码内注释充足，但缺少"活代码 vs 死代码"的区分 |

### 5.4 结论

**当前代码对于"经典五级流水线性能模拟"这一目标已经满足需求。** 核心机制的实现是正确且完整的——转发网络、Load-Use 停顿、IPC 统计都经过了 30 个测试的验证。

**存在的工程问题**（不影响当前功能，但值得后续关注）：
1. LSU 对象实例化但完全未接入——要么接入，要么移除实例化代码
2. `HazardResult::fwd_src*` 是运行时死代码——要么接入到提交阶段做运行时断言，要么仅保留为测试接口并明确注释
3. `flush_if` / `flush_id` 预留了但从未激活——这是为分支预测预留的，符合"待做"列表
4. Load 的 `mem_addr → alu_result` 路径是死数据路径——当 Cache 建模时需激活

**Trace 驱动模式的根本性限制**（非本文档范围内可解决）：
- 无分支预测 → 高估 IPC（对分支密集型程序影响显著）
- 无 Cache 建模 → 无法评估存储系统性能
- 无多核同步 → 无法扩展到多核（需要 Trace 同步注解，参见 `docs/resc/(0504)关于TraceDriven能做到多少.md`）
