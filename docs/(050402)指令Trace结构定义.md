# 指令Trace结构定义

为了让 Trace-Driven Simulator 能够顺序执行，并为未来的乱序执行、流水线级和依赖检查预留接口，我们需要提取并固化核心的指令信息，而抛弃非必要的信息。

## 1. 结构定义方案

在 Rust 模拟器中，每条动态指令对应一个 `TraceRecord`，其数据结构设计如下：

```rust
#[repr(u8)]
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum OpType {
    Unknown = 0,
    IntAlu  = 1, // 算术逻辑运算
    Load    = 2, // 内存读取
    Store   = 3, // 内存写入
    Branch  = 4, // 分支指令
    Jump    = 5, // 跳转指令 (JAL, JALR)
    System  = 6, // 系统指令 (ECALL, CSR 等)
}

#[derive(Debug, Clone, Copy)]
pub struct TraceRecord {
    pub pc: u64,           // 指令所在的 PC
    pub inst: u32,         // 32 位指令机器码原始值

    // 预解码的寄存器信息（用于快速的依赖检查机制，0 表示不使用或 zero 寄存器）
    pub rs1: u8,
    pub rs2: u8,
    pub rd:  u8,

    pub op_type: OpType,   // 预解码的指令类型

    // Spike 记录的实际副作用信息 (供验证执行正确性或用于Cache访存模拟)
    pub mem_addr: u64,     // 访存地址 (Load/Store时有效)
    pub rd_val: u64,       // 写入目标寄存器的实际值
}
```

### 为什么这么设计？
1. **摒弃解析汇编字符串**：不用在模拟器里存 `addi a0, a0, 10` 等字符串，节省内存。通过 `inst` (如 `0x00a50513`)，我们可以在模拟器需要的时候随时做完整的重解（Decode）。
2. **预解码提取依赖 (rs1, rs2, rd, op_type)**：流水线模拟或记分板（Scoreboard）最核心的就是依赖检查，直接在 Trace 中暴露出这些字段，省去在热路径中反复 decode 的开销。
3. **真实访存和写回 (mem_addr, rd_val)**：由 Spike 提供的执行真值。即便我们的模拟器不真正做物理 ALU 运算（或者算错），也可以依靠这些值直接更新体系结构状态，专注于“时序”和“依赖”的模拟。

## 2. 二进制存储格式 (.trace)

为了读取的高效性，我们将这组结构以二进制方式存储。单个 `TraceRecord` 占用极小的空间（比如按 `pc(8) + inst(4) + rs1(1) + rs2(1) + rd(1) + op(1) + mem_addr(8) + rd_val(8) = 32 Byte` 对齐）。
模拟器启动时，直接可以通过 `std::slice::from_raw_parts`（或 `zerocopy`、`bytemuck` 等库）把文件 mmap 进内存，实现极速加载。
